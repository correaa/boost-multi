\frametitle{As a proposal}

Why not proposed for Standard?
\begin{itemize}
  \item Can't promise ABI compatibility
  \item Controversial idioms (some hacks)
  \item Too restrictive for compatibility with exotic 3rd parties (Thrust, BLAS, CUDA, cuLibs)
  \item Enough overlap with \texttt{std::span} and \texttt{std::mdarray}
  \item Long time horizon
\end{itemize}
 
Why proposed to Boost?
\begin{itemize}
  \item Ideal balance between risk (innovation) and requirements (quality, guidelines)
  \item \emph{Peer reviewed}
  \item Still good distribution model and visibility
  \item No need for compiler-magic (although new language features would help)
  \item Moderate time horizon
  \item Can eventually benefit from low level Boost tools (Boost.Hana, Boost.Assert, Boost.Align, Boost.STLInterfaces) 
  \item Peer interaction with Boost.Serialization, Boost.JSON, Boost.QVM, Boost.YAP
\end{itemize}

\begin{frame}[fragile]{General lazy arrays: function restricted to a domain}

A \lstinline|multi::restriction| 
is the restriction (in the mathematical sense) of a function to a bounded index domain.

\begin{lstlisting}
// lazy 2x3 array R[i][j] = i + j, no memory allocated
auto R = multi::restriction({2, 3}, [](auto i, auto j) { return i + j; });

// equivalent idiomatic form using the ^ operator ("restricted to")
auto R = [](auto i, auto j) { return i + j; } ^ multi::extensions_t{2, 3};

R[1][2];           // 3         (computed on access)
R.size();          // 2
R.num_elements();  // 6
+R;                // materialize into multi::array<int, 2>
\end{lstlisting}

\medskip
Models the Multi array concept --- it has \lstinline|extents|, iterators, \lstinline|operator[]|, etc.\\
Structural operations compose (still) without allocating: \lstinline|transposed|, \lstinline|repeated|, \lstinline|diagonal|, \lstinline|partitioned|, \lstinline|reversed|, \lstinline|rotated|, \lstinline|transformed|, \lstinline|element_transformed|.
\end{frame}

\begin{frame}[fragile]{\lstinline|multi::restriction|: composition}

\begin{lstlisting}
auto iota = [](multi::index i) { return i; } ^ multi::extensions_t<1>(5);
// iota   :  {0, 1, 2, 3, 4}

auto M = iota.repeated(4);       // lazy 4x5, all rows equal to iota
auto D = ([](auto i, auto j) { return i*i + j*j; }
         ^ multi::extensions_t<2>(3, 3)).diagonal();   // {0, 2, 8}

// softmax via lazy restrictions (no temporaries)
using multi::elementwise::operator-; using multi::elementwise::operator/;
auto x = exp(~mat - (mat | maxR1));   // row-wise subtract max
auto y = ~(x / ((~x) | sumR1));       // normalize
\end{lstlisting}

\medskip
\begin{itemize}
  \item A \lstinline|restriction| is essentially a lambda + an \lstinline|extensions_t<D>|.
  \item Passes through \lstinline|transposed|, \lstinline|diagonal|, etc.\ by rebinding arguments.
  \item Composable with owning \lstinline|multi::array| via elementwise operators.
  \item Materialize with \lstinline|+R| or assignment into an \lstinline|array|.
\end{itemize}

\end{frame}


\begin{frame}[fragile]{Aliasing, and how functional idioms partially solve it}

Classic pitfall: two pointers/references may refer to the \emph{same} memory; the compiler must assume they do, and optimizations (vectorization, reordering, fusion) are disabled.

\begin{lstlisting}
multi::array<double, 1> y = /* ... */;
y = M * y;               // self-aliasing: y is both source and destination
A = A.transposed();      // writes overlap with reads
\end{lstlisting}

Usual patches --- C's \lstinline|restrict|, \lstinline|__restrict__|, expression templates with bookkeeping --- are brittle and non-portable. A value-based, pointer-free expression \emph{cannot} alias by construction.

\begin{lstlisting}
auto R = [](auto i, auto j) { return i + j; } ^ multi::extensions_t{N, N};
auto T = R + R.transposed();          // lazy: no memory, no pointers

multi::array<int, 2> out = T;         // single write into a fresh array
\end{lstlisting}

\begin{itemize}
  \item No intermediate buffers \(\Rightarrow\) no aliasing between stages.
  \item Materialization is the \emph{only} write --- compiler sees a pure nested loop.
  \item \emph{Partial}: assigning back into an array whose storage backs the RHS still aliases; lambda captures-by-reference can re-introduce it.
\end{itemize}

\end{frame}

\begin{frame}[fragile]{Out-of-bounds pointer formation in strided iteration}

Boost.Multi provides STL-compatible strided iterators --- \lstinline|.begin()|, \lstinline|.end()|, range-for loops.\\
But computing \lstinline|.end()| can form a pointer \emph{past} the allocation:

\begin{lstlisting}
multi::array<double, 1> A(10);   // 10-element allocation [base, base+10)

auto strided = A.strided(3);    // elements {0, 3, 6, 9}
strided.end();                   // computes base + 3*4 = base + 12
                                 // 2 past the allocation: UB under [expr.add]
\end{lstlisting}

Similarly: \lstinline|.reversed()| forms a pointer \emph{before} the allocation; \lstinline|.transposed()| iterators may overshoot when columns $\times$ row-stride $>$ total elements.

\medskip
\begin{itemize}
  \item The UB is in \emph{forming} the pointer, not dereferencing it.
  \item Index-based access (\lstinline|A[i][j]|, \lstinline|A(i,j)|) is \emph{not} affected.
  \item \lstinline|multi::restriction|s are not affected --- no pointer arithmetic involved.
  \item No compiler currently exploits this, but \lstinline|constexpr| and sanitizers can flag it.
  \item Other libraries (\lstinline|std::mdspan|, Eigen, Kokkos) avoid this by not providing strided iterators at all.
\end{itemize}

\end{frame}


\begin{frame}[fragile]{Algorithm Dispatching}{
How does the library call the right \texttt{copy\_n} for CPU vs GPU?
}


\medskip
\begin{lstlisting}
class adl_copy_n_t {
  auto _(priority<0>, auto... as) ... {return std::copy_n(...);}    // CPU fallback
  auto _(priority<1>, auto... as) ... {return thrust::copy_n(...);} // GPU
  auto _(priority<2>, auto... as) ... {return copy_n via(...);}     // ADL
  auto _(priority<3>, auto it, auto... rst) ... {return it.copy_n(...); } // member
public:
  template<class... As>
  auto operator()(As&&... as) const
    -> decltype(...) { return _(priority<3>(), std::forward<As>(as)...)); }
};
inline constexpr adl_copy_n_t adl_copy_n;
\end{lstlisting}

\medskip
\begin{itemize}
  \item Overload resolution picks the \emph{highest priority} that compiles (SFINAE)
  \item Same call site works for CPU (\lstinline|std::|), GPU (\lstinline|thrust::|), or user types
  \item Extensible (non-intrusive but dangerous): add a \lstinline|copy_n| free function found via ADL
  \item Extensible (intrusive): add members: add a \lstinline|copy_n| member function
\end{itemize}

\end{frame}

\begin{frame}[fragile]{OpenMP via Thrust Backend}

Use Thrust's OMP backend for parallel CPU execution --- no GPU required:

\begin{lstlisting}
#include <boost/multi/adaptors/thrust/omp.hpp>

// array backed by thrust::omp::allocator
multi::thrust::omp::array<double, 1> arr(64);  // <double, 1, thrust::omp::allocator<double>>

// parallel fill with OpenMP
#pragma omp parallel for
for(int i = 0; i < arr.size(); ++i)
  arr[i] = double(i) * double(i);

// thrust::reduce with OMP execution policy
auto sum = thrust::reduce(thrust::omp::par,
  arr.begin(), arr.end(), 0.0);
\end{lstlisting}

\begin{itemize}
  \item \lstinline|multi::thrust::omp::array<T, D>| = \lstinline|multi::array<T, D, thrust::omp::allocator<T>>|
  \item Thrust algorithms dispatch to OpenMP automatically via allocator tag
  \item Same code structure as CUDA --- swap allocator to switch backend
\end{itemize}

\end{frame}

\begin{frame}[fragile]{The Problem: Row-wise Reduction}

Given a 2D array, reduce each row independently:
\begin{lstlisting}
multi::array<double, 2> M({100, 1000});
auto row_sums = multi::array<double, 1>(M.size(0));

// Better: use transform instead of raw loop
thrust::transform(M.begin(), M.end(), row_sums.begin(),
  [](auto const& row) {
    return thrust::reduce(row.begin(), row.end());
  }
);
\end{lstlisting}

\textbf{Problem:} Even with transform, each row reduction still happens one at a time — the reductions are \emph{serialized}
\end{frame}

\begin{frame}[fragile]{Solution: reduce\_by\_index}

Use Thrust's built-in parallelism:
\begin{lstlisting}
auto row_sums = multi::reduce_by_index(M);
// Massively parallel! All rows reduce simultaneously
\end{lstlisting}

\textbf{How it works:}
\begin{itemize}
  \item Maps each element to its row index
  \item Uses \lstinline|thrust::reduce_by_key| (GPU-native)
  \item Parallelizes across all rows at once
\end{itemize}

\textbf{Speedup:} Parallelization across all \(N\) rows vs. serialized $N$ iterations
\end{frame}

\begin{frame}[fragile]{Custom Binary Operations}

\begin{lstlisting}

\begin{frame}[fragile]{Adaptor example: BLAS}

\begin{lstlisting}
#include <boost/multi/adaptors/blas.hpp>

multi::array<double, 2> A({3, 3}), B({3, 3}), C({3, 3});
// ...
namespace blas = boost::multi::blas;
blas::gemm(1.0, A, B, 0.0, C);   // C = A * B
auto At = blas::T(A);             // lazy transpose view
auto nrm = blas::nrm2(A[0]);      // L2 norm of a row
\end{lstlisting}

\begin{itemize}
  \item Works with CPU BLAS, cuBLAS, and rocBLAS --- same code, different allocator
  \item Lazy \lstinline|blas::T(A)| and \lstinline|blas::H(A)| avoid copies
\end{itemize}
